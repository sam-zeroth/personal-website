Building a 3D experience for the web means fighting for every frame. Unlike native apps or games, you're competing with browser overhead, garbage collection pauses, and users on five-year-old laptops. Here's what I learned building the brain visualization on this site.

## Start with the budget

Before writing any Three.js code, decide your performance budget:

- **Target frame rate**: 60fps on modern devices, 30fps on mobile
- **Draw calls**: Under 50 for complex scenes
- **Triangle count**: Under 100K for web
- **Texture memory**: Under 64MB

These numbers seem low if you're coming from game dev. They're reality for the web.

## Instancing is your best friend

The single biggest performance win in the brain visualization was switching from individual meshes to `InstancedMesh`. Instead of creating 2,000 separate neuron objects:

```typescript
// Before: 2,000 draw calls 💀
neurons.forEach(pos => {
  const mesh = new THREE.Mesh(geometry, material);
  mesh.position.copy(pos);
  scene.add(mesh);
});

// After: 1 draw call ✨
const instancedMesh = new THREE.InstancedMesh(
  geometry,
  material,
  neurons.length
);

const matrix = new THREE.Matrix4();
neurons.forEach((pos, i) => {
  matrix.setPosition(pos);
  instancedMesh.setMatrixAt(i, matrix);
});
```

This took the scene from 15fps to 60fps on my test devices. One change. That's the power of batching.

## Custom shaders over material properties

Three.js's built-in materials are convenient but generic. For the neuron glow effect, a custom shader is both prettier and faster:

```glsl
// Fragment shader for neuron glow
varying float vIntensity;
uniform vec3 uColor;
uniform float uTime;

void main() {
  float pulse = sin(uTime * 2.0 + vIntensity * 6.28) * 0.3 + 0.7;
  float alpha = smoothstep(0.0, 1.0, vIntensity) * pulse;
  gl_FragColor = vec4(uColor, alpha * 0.6);
}
```

Custom shaders let you move computation to the GPU where it belongs. The CPU just updates a single `uTime` uniform each frame.

## Dispose of everything

Three.js doesn't garbage collect GPU resources. If you create a geometry, material, or texture, you own it until you explicitly call `.dispose()`:

```typescript
useEffect(() => {
  return () => {
    geometry.dispose();
    material.dispose();
    texture?.dispose();
  };
}, []);
```

In React Three Fiber, this is mostly handled for you — but if you're creating resources in `useMemo` or `useRef`, you still need cleanup.

## Mobile-specific tricks

Mobile GPUs are a different beast. A few things that helped:

- **Reduce pixel ratio**: `renderer.setPixelRatio(Math.min(window.devicePixelRatio, 2))` — nobody needs 3x on a phone
- **Simplify on detect**: Check `navigator.hardwareConcurrency` and reduce particle counts for low-end devices
- **Avoid transparency sorting**: Use `depthWrite: false` and additive blending instead of alpha blending where possible

## The result

After all optimizations, the brain scene renders at 60fps on a MacBook Air and 45-55fps on a mid-range Android phone. The total JavaScript bundle for the 3D scene is under 80KB gzipped.

Performance isn't about making things less beautiful. It's about being intentional with where you spend your GPU budget.
